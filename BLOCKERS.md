## 0. 上一轮云端任务失败根因（已修复，无需你操作）

**背景**：上一轮云端任务（即本分支 `trae/agent-NG1qVv` 的首个提交）在收尾时"卡住失败"，根本原因是 **Git LFS 配置与实际存储状态不一致**：`.gitattributes` 声明 `*.jpg/*.jpeg/*.png/*.mp3/*.mp4` 走 LFS，但 master 上这些文件实际是普通二进制 blob（从未做 `git lfs migrate`）。上一轮任务 `git add` 改动过的 6 个文件（5 张图 + `lovebgm.mp3`）被 LFS clean filter 转成了 132 字节的**指针文件**提交进分支，而 LFS 对象只留在沙箱本地 `.git/lfs/objects`、**没推到 GitHub LFS store**。结果：任何人检出本分支，这 6 个文件都是坏的指针，求婚页背景乐、走婚桥/金佛山/马尿水瀑布等照片全部损坏。更糟的是两个部署 workflow（GitHub Pages / 魔搭同步）都没安装 git-lfs，即便对象推了，生产构建也只会拿到指针。

**已修复**：
1. 移除 `.gitattributes` 中 `*.jpg/*.jpeg/*.png/*.mp3/*.mp4` 的 LFS 规则（保留魔搭默认的 ML 模型规则，本项目无此类文件），并加注释说明不可重加的原因。这与 `sync-to-modelscope.yml` 注释里的"未走 LFS"设计意图一致。
2. 从 master 恢复 6 个真实二进制到本分支，已校验索引中 blob 大小（7.4MB / 375KB / 320KB / 210KB / 220KB / 11.7MB）与原始一致。
3. 顺带修了根 `index.html` 首屏"求婚 · 正在进行"的过期文案 → "求婚 · 2026.02.14"（求婚已于 2026-02-14 完成，见 `milestones.json`/`site.json`）。

**大包 push 兜底**：魔搭同步的大包 "curl 18" 由 `sync-to-modelscope.yml` 的 `http.postBuffer=524288000` + 5 次指数退避重试 + 浅拉合并兜底，无需 LFS。本仓库图片约 117MB + 音频 12MB，在 postBuffer 500MB 内，可正常推送。
